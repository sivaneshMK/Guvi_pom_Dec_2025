Bug life cycle
---------------
Bug Life Cycle
In software development process, the bug has a life cycle. The bug should go through the life cycle to be closed A specific life cycle ensures that the process is standardized The bug attains different states in the life cycle The life cycle of the bug can be shown diagrammatically as follows

New:
When a bug is found/revealed for the first time, the software tester communicates it to his/her team leader (Test Lead) in order to confirm if that is a valid bog. After getting confirmation from the Test Lead, the software tester logs the bug and the status of New is assigned to the bug

Assigned
After the bug is reported as New it comes to the Development Team. The development Lead verifies if the bug is valid. If the Bug is valid, development leader assign it to developer to fix it and a status of 'Assigned is assigned to it

InProgress:
Once the developer starts working on the bug. he/she changes the status of the bug to InProgress to indicate that he/she is working on it to find a solution.

Open:
Once development lead is verified the bug but it not assigned to any of the developers to fix, because of developers may be busy with other bug or task, that time the bug is moved to the open status. Once the developer get free he will assign the bug to his/her name and changing the status to inprogress.

Fixed
Once the developer makes necessary changes in the code and verifies the code, he/she marks the bug as Fixed and passes it over to the Development Lead in order to pass it to the testing team.

Retest:
The testing team leader changes the status of the bug, which is previously marked with fixed to 'Retest' and assigns it to a tester for retesting

Closed:
After the bug is assigned a status of 'Retest, it is again tested. If the problem is solved. the tester closes it and marks it with 'Closed' Status.

Reopen:
If after retesting the software for the bug opened, if the system behaves in the same way or same bug arises once again, then the tester reopens the bug and again sends it back to the developer marking its status as 'Reopenâ€™.

Rejected:
If the Testing Lead finds that the system is working according to the specification on the bug is invalid as per the explanation from the development, he/she add the  log and marks in status as Rejected.

Postponed
Sometimes, testing of a particular bug has to be postponed for an indefinite period. This station may occur because of many reasons, such as unavailability of Test data, unavailability of particular functionality etc. That time, the bug is marked with Postponed status.

Deferred:
In some cases a particular bug stands no importance and needs to be avoided that time it is marked with Deferred status.

pipeline{
    
    agent any
    stages{
        
        stage('Check Python'){
            steps{
                bat 'python --version'
            }
            
        }
        
        stage('Create Virtual Environment'){
            steps{
                bat 'python -m venv venv'
            }
        }
        stage('Checkout Code'){
            steps{
                git 'https://github.com/sivaneshMK/Guvi_pom_Dec_2025.git'
            }
        }
        
        stage('Activate & install Dependencies'){
            steps{
                bat '''
                call venv\\Scripts\\activate
                pip install --upgrade pip
                pip install -r requirements.txt
                '''
            }
        }
        
        stage('Run Pytest'){
            steps{
                bat '''
                call venv\\Scripts\\activate
                pytest --env=test --html=report.html --self-contained-html'''
            }
        }
    }
        post{
            always{
                archiveArtifacts artifacts: 'report.html', fingerprint:true
            }
        }
    }
